# ComfyUI-omni-llm 节点参数说明与推荐设置

本文档详细介绍了 ComfyUI-omni-llm 插件中各个节点的参数作用和推荐设置，帮助用户更好地使用和调节模型。

## 动态模型支持

本插件实现了智能模型检测系统，能够自动发现并支持llama_cpp_python新增的VL模型。当llama_cpp_python更新并支持新的模型版本时，用户可直接下载使用，无需修改插件代码。

## 文档结构

本文档按照以下结构组织：
- **节点参数说明**：详细介绍每个节点的参数及其作用
- **设备性能适配建议**：基于不同硬件配置的推荐设置
- **支持的模型说明**：各种模型的特点和适用场景
- **常见问题与解决方案**：使用过程中可能遇到的问题及解决方法
- **使用建议与最佳实践**：提高模型性能和结果质量的技巧

## 1. 模型加载节点

### 1.1 分段模型加载器 (llama_cpp_sharded_model_loader)

### 1.2 Llama-cpp 模型加载节点 (llama_cpp_model_loader)

### 功能
加载和初始化分段模型（如Qwen2.5-Omni系列），支持自动检测本地模型文件，是分段模型推理的基础节点。

### 参数说明

#### 基础参数

参数名：**模型名称** 
- 作用说明：选择要加载的分段模型
- 推荐设置：根据需要选择合适的分段模型 
- 支持格式：分段.safetensors文件
- 示例：
  - Qwen2.5-Omni-3B（默认）
  - Qwen2.5-Omni-7B

- 自动检测：本地模型会自动扫描并显示为"Local: 模型名称"

参数名：**量化模式 (quantization)**
- 作用说明：选择模型量化模式
- 选项：
  - `None`：不量化（精度最高，显存占用最大）
  - `4-bit (VRAM-friendly)`：4位量化（显存友好，适合显存有限的GPU）
  - `8-bit (Balanced Precision)`：8位量化（平衡精度和显存占用）
- 推荐设置：
  - GPU模式：根据显存选择4-bit或8-bit
  - CPU模式：建议选择None（CPU模式下量化基本无效）
- 注意事项：CPU模式下量化可能导致性能下降或加载失败

#### 运行模式参数

以下参数与常规模型加载节点完全相同，请参考**1. Llama-cpp 模型加载节点**部分的详细说明：

- **启用多模态功能 (enable_mmproj)**
- **启用ASR (enable_asr)**
- **启用TTS (enable_tts)**
- **运行模式 (device_mode)**
- **上下文长度 (n_ctx)**
- **GPU模型层数 (n_gpu_layers)**
- **显存限制 (vram_limit)**
- **图片最小编码token数 (image_min_tokens)**
- **图片最大编码token数 (image_max_tokens)**
- **注意力类型 (attention_type)**

### 特殊说明

#### 模型加载特点

1. **自动模型检测**：自动扫描LLM文件夹中的分段模型文件
2. **量化兼容性**：支持4-bit和8-bit量化，但CPU模式下效果有限
3. **硬件适配**：根据硬件性能自动调整默认参数
4. **模型完整性检查**：自动验证模型文件是否完整

#### 使用建议

1. **模型选择**：
   - 低显存设备：Qwen2.5-Omni-3B
   - 中等显存设备：Qwen2.5-Omni-7B


2. **量化选择**：
   - GPU模式：根据显存大小选择合适的量化级别
   - CPU模式：建议不使用量化，直接使用None

3. **参数优化**：
   - 首次使用时建议使用默认参数
   - 根据硬件性能逐步调整上下文长度和GPU层数
   - 遇到显存不足时，优先调整量化模式和GPU层数

## 1.2 Llama-cpp 模型加载节点 (llama_cpp_model_loader)

### 功能
加载和初始化 LLM（大语言模型）/VLM（视觉语言模型）模型，是所有其他节点的基础。支持.gguf和.safetensors格式，自动检测多分段模型。

### 参数说明

#### 基础参数

参数名：**模型文件** 
- 作用说明：选择要加载的 LLM 模型文件 
- 推荐设置：根据需要选择合适的 GGUF 模型 
- 支持格式：.gguf, .safetensors（含多分段）
- 示例：
  - 轻量级任务：Moondream2、MobileVLM
  - 中型任务：LLaVA-1.6、MiniCPM-V-4.5
  - 复杂任务：Qwen3-VL、GLM-4.6V
- 自动排除：自动过滤TTS/ASR语音模型、mmproj视觉编码模型

参数名：**启用多模态功能 (enable_mmproj)**
- 作用说明：启用后自动选择对话格式处理器并启用多模态功能
- 推荐设置：处理图片或需要对话格式时设为 True，纯文本生成时设为 False
- 注意：启用后需要选择对应的视觉编码模型

参数名：**视觉编码模型 (mmproj)**
- 作用说明：选择对应的视觉编码模型文件
- 推荐设置：启用多模态功能时选择匹配的模型
- 支持格式：.gguf, .safetensors（含多分段）
- 注意事项：
  - 不同模型需要对应的 mmproj 文件，确保版本匹配
  - 多分段模型显示为"model-of-* (多分段)"
  - 支持"mmproj"或"vision"命名的视觉编码模型

参数名：**启用ASR (enable_asr)**
- 作用说明：启用ASR语音识别功能
- 推荐设置：需要语音识别时设为 True
- 前提条件：需配合ASR模型加载器使用

参数名：**启用TTS (enable_tts)**
- 作用说明：启用TTS语音合成功能
- 推荐设置：需要语音合成时设为 True
- 前提条件：需配合TTS模型加载器使用

#### 运行模式参数

参数名：**运行模式 (device_mode)**
- 作用说明：选择运行模式：GPU=使用显卡加速，CPU=纯CPU运行
- 选项：GPU / CPU
- 推荐设置：
  - 8GB+显存：GPU（推荐）
  - <8GB显存：CPU
  - 无 GPU：CPU
- 智能默认：系统根据硬件自动推荐

参数名：**上下文长度 (n_ctx)**
- 调节范围：1024-327680
- 作用说明：上下文长度，影响可处理的文本长度
- 推荐设置：
  - 24GB+显存（5090/4090）：16384
  - 16GB显存（4080）：8192
  - 12GB显存（4070 Ti/3080）：6144
  - 8GB显存（3070/3060）：4096
  - 4-6GB显存：2048
- 特殊模型要求：
  - Qwen3系列：GPU模式建议≤4096，CPU模式建议≤2048
  - MiniCPM-o-4.5：GPU模式建议≤4096，CPU模式建议≤2048
- 经验法则：上下文长度与任务复杂度成正比，与硬件性能成反比

参数名：**GPU模型层数 (n_gpu_layers)**
- 调节范围：-1-1000
- 作用说明：加载到GPU的模型层数，-1=全部加载（GPU模式有效）
- 推荐设置：
  - GPU 模式：
    - 24GB+显存（5090/4090）：-1（全部加载）
    - 16GB显存（4080）：-1（全部加载）
    - 12GB显存（4070 Ti/3080）：-1（全部加载）
    - 8GB显存（3070/3060）：30（部分加载）
  - CPU 模式：自动忽略，无需设置

参数名：**显存限制 (vram_limit)**
- 调节范围：-1-24
- 作用说明：显存限制（GB），-1=无限制（GPU模式有效）
- 推荐设置：
  - GPU 模式：通常设为 -1 或实际显存大小减1（16g就设置成15）
  - CPU 模式：自动忽略，无需设置

#### 图像处理参数

参数名：**图片最小编码token数 (image_min_tokens)**
- 调节范围：0-4096
- 作用说明：图片最小编码token数
- 推荐设置：
  - 默认：0（自动）
  - Qwen3系列：≥1024（自动调整）
  - MiniCPM-o-4.5：≥768（自动调整）
- 注意事项：某些模型需要最小token数才能正常处理图像

参数名：**图片最大编码token数 (image_max_tokens)**
- 调节范围：0-4096
- 作用说明：图片最大编码token数
- 推荐设置：保持默认 0（自动）
- 注意事项：如设置，必须≥image_min_tokens

#### 高级参数

参数名：**注意力类型 (attention_type)**
- 调节范围：Auto/Standard/Flash/XFormers
- 作用说明：选择模型使用的注意力机制类型，影响模型推理速度和内存使用
- 推荐设置：Auto（默认）
- 选项说明：
  - Auto：自动选择（推荐，NVIDIA GPU自动启用Flash Attention）
  - Standard：标准注意力机制（兼容性最好，适用于所有硬件）
  - Flash：Flash Attention（NVIDIA GPU专用优化，推理速度最快）
  - XFormers：XFormers优化（特定场景下内存使用更高效）
- 使用场景：遇到注意力机制兼容性问题或需要优化性能时手动调整

### 智能推荐
- 系统会根据硬件性能自动推荐 device_mode、n_ctx 和 n_gpu_layers 的默认值
- 低性能设备会自动降低默认参数以保证运行流畅
- CPU 模式会自动忽略 GPU 相关参数，无需手动调整
- 批处理参数（n_batch, n_ubatch, n_threads, n_threads_batch）已改为自动根据硬件性能调整

### 1.3 CPU/GPU 运行模式详解

#### device_mode 参数说明

`device_mode` 参数允许用户选择模型的运行模式，提供了灵活的硬件适配方案。

#### GPU 模式（推荐）

**适用场景**：
- 有可用 GPU 显存
- 需要快速推理
- 处理大型模型或复杂任务

**特点**：
- 推理速度快，适合实时应用
- 支持更大的模型和更长的上下文
- 自动进行显存估算和优化
- 根据显存大小智能调整参数

**参数设置**：
- `n_gpu_layers`：控制加载到 GPU 的模型层数，-1 表示全部加载
- `vram_limit`：显存限制（GB），-1 表示无限制
- 批处理参数（n_batch, n_ubatch, n_threads, n_threads_batch）已改为自动根据硬件性能调整

**推荐配置**：
- 24GB+ 显存：n_gpu_layers = -1, vram_limit = 24
- 16GB 显存：n_gpu_layers = -1, vram_limit = 16
- 12GB 显存：n_gpu_layers = -1, vram_limit = 12
- 8GB 显存：n_gpu_layers = 30, vram_limit = 8

**性能表现**：
- 推理速度：快（取决于 GPU 性能）
- 显存占用：高（根据模型大小）
- 适用任务：所有类型，特别是实时应用

#### CPU 模式

**适用场景**：
- 无 GPU 或 GPU 显存不足
- 需要使用 CPU 进行推理
- 低性能硬件（<8GB 显存）
- 需要最大兼容性

**特点**：
- 不依赖 GPU 显存
- 自动忽略 GPU 相关参数
- 推理速度较慢，但兼容性好
- 适用于所有硬件配置

**参数设置**：
- CPU 模式下，n_gpu_layers 和 vram_limit 参数会被自动忽略
- 无需手动调整这些参数
- 系统会自动设置 n_gpu_layers = 0, vram_limit = -1

**推荐配置**：
- 适用于所有硬件配置
- 适合小型模型和简单任务
- 显存不足时的备选方案

**性能表现**：
- 推理速度：慢（取决于 CPU 性能）
- 显存占用：低（仅使用系统内存）
- 适用任务：简单任务、离线处理

#### 智能默认值

插件会根据硬件性能自动选择合适的运行模式：

- **高性能硬件**（8GB+ 显存）：默认使用 GPU 模式
- **低性能硬件**（<8GB 显存）：默认使用 CPU 模式
- **无 GPU 检测**：默认使用 CPU 模式

#### 使用建议

1. **优先使用 GPU 模式**：
   - 如果 GPU 显存充足，优先选择 GPU 模式以获得更好的性能
   - GPU 模式推理速度通常比 CPU 模式快 5-10 倍

2. **显存不足时切换 CPU**：
   - 如果遇到显存不足错误（OOM），可以尝试切换到 CPU 模式
   - CPU 模式虽然慢，但不会受显存限制

3. **灵活切换**：
   - 可以根据任务需求随时切换运行模式
   - 不同任务可以使用不同的运行模式

4. **监控性能**：
   - 使用 GPU 模式时，注意监控显存使用情况
   - 使用任务管理器或 GPU 监控工具查看显存占用

5. **参数优化**：
   - GPU 模式：可以调整 n_gpu_layers 和 vram_limit 来优化性能
   - CPU 模式：无需调整 GPU 相关参数，专注于其他推理参数

#### 常见问题

**Q: 什么时候应该使用 CPU 模式？**
A: 当遇到以下情况时建议使用 CPU 模式：
- 没有可用的 GPU
- GPU 显存不足以加载模型
- 需要处理多个大型模型，显存不足
- 需要最大兼容性

**Q: GPU 模式和 CPU 模式的性能差异有多大？**
A: 通常情况下，GPU 模式的推理速度比 CPU 模式快 5-10 倍，具体取决于硬件配置和模型大小。

**Q: CPU 模式下 n_gpu_layers 参数还有作用吗？**
A: 没有。CPU 模式会自动忽略 n_gpu_layers 和 vram_limit 参数，强制使用纯 CPU 运行。

**Q: 如何知道当前使用的是 GPU 还是 CPU 模式？**
A: 查看控制台日志输出，会显示当前使用的设备模式：
- GPU 模式：`【设备模式】使用 GPU 模式（n_gpu_layers=X, vram_limit=XGB）`
- CPU 模式：`【设备模式】使用 CPU 模式（忽略 n_gpu_layers 和 vram_limit 参数）`

**Q: 可以在运行时动态切换 GPU 和 CPU 模式吗？**
A: 可以。每次重新加载模型时，可以切换 device_mode 参数。但需要注意的是，切换模式会重新加载模型，可能需要一些时间。

### 1.4 性能优化参数详解

本插件提供了多种性能优化参数，可以根据硬件配置和使用场景进行精细调整，以获得最佳的性能和显存使用效率。

#### 自动调整的批处理参数

以下参数已改为根据硬件性能自动调整，无需手动设置：

- **n_batch（批处理大小）**：根据硬件性能等级自动设置（1024-4096）
- **n_ubatch（物理批处理大小）**：自动设置为 n_batch 的 1/4
- **n_threads（生成线程数）**：根据 CPU 核心数自动设置
- **n_threads_batch（批处理线程数）**：根据 CPU 核心数自动设置

#### 硬件配置参数推荐表

| 硬件配置 | n_batch | n_gpu_layers | vram_limit | n_ctx |
|---------|--------|-------------|-----------|------|
| 高性能硬件（24GB+ 显存） | 4096 | -1 | 24 | 16384 |
| 中高配硬件（16GB 显存） | 3072 | -1 | 16 | 8192 |
| 中配硬件（12GB 显存） | 2048 | -1 | 12 | 6144 |
| 中低配硬件（8GB 显存） | 1536 | 30 | 8 | 4096 |
| 低配硬件（<8GB 显存） | 1024 | 0 | -1 | 2048 |

#### 性能优化配置示例

##### 高性能配置（24GB+ 显存）
```
n_gpu_layers: -1
vram_limit: 24
```

##### 中等性能配置（12-16GB 显存）
```
n_gpu_layers: -1
vram_limit: 16
```

##### 低性能配置（8GB 显存）
```
n_gpu_layers: 30
vram_limit: 8
```

##### CPU 模式配置
```
device_mode: CPU
```
说明：CPU 模式下 GPU 相关参数自动忽略

#### GPU 厂商适配
- **NVIDIA GPU**：Flash Attention 已自动启用
- **AMD GPU**：批处理参数已自动优化
- 不同 GPU 需要不同的优化策略

#### 性能优化技巧

1. **优先调整显存相关参数**：
   - 调整 `n_gpu_layers`（GPU模型层数）
   - 调整 `vram_limit`（显存限制）

2. **逐步优化**：
   - 从默认配置开始
   - 每次只调整一个参数
   - 观察性能变化，找到最佳配置

3. **监控资源使用**：
   - 使用任务管理器或 GPU 监控工具
   - 观察显存占用和推理速度
   - 根据实际情况调整参数

4. **平衡性能和质量**：
   - 显存不足时，优先保证可用性
   - 显存充足时，优先追求性能
   - 根据任务需求灵活调整

## 2. Llama-cpp 统一推理节点 (llama_cpp_unified_inference)

### 功能
进行 LLM/VLM/Omni 模型推理，支持纯文本、单张图片、多张图片、音频和视频输入，是统一的推理入口。

**注意**：此节点支持所有模型类型（LLM、VLM、Omni），无需区分模型类型。

### 参数说明

#### 模型配置

参数名：**Llama模型**
- 作用说明：选择加载的LLM/VLM/Omni模型，用于推理处理
- 推荐设置：连接到模型加载节点
- 支持：所有模型类型（LLM、VLM、Omni）

#### 推理模式

参数名：**推理模式 (inference_mode)**
- 作用说明：选择推理模式，决定节点处理的输入类型和任务类型
- 选项：
  - `[基础] 文本生成 (Text Generation)`：纯文本模式，仅处理文本输入
  - `[基础] 图像理解 (Image Understanding)`：图像处理模式，处理单张或多张图片
  - `[基础] 音频转文本 (Audio to Text)`：使用ASR模型将音频转换为文本
  - `[基础] 文本转音频 (Text to Audio)`：生成文本后使用TTS模型转换为语音
  - `[高级] 视频理解 (Video Understanding)`：处理视频文件，提取帧进行分析
  - `[高级] 全模态整合 (Multimodal Integration)`：同时处理文本、图像、音频和视频的复杂多模态任务
- 推荐设置：
  - 纯文本任务：`[基础] 文本生成` - 适合聊天、写作、问答等纯文本场景
  - 单张图片分析：`[基础] 图像理解` - 适合图片描述、物体识别、OCR等任务
  - 音频处理：`[基础] 音频转文本` - 适合语音识别、音频内容提取等任务
  - 语音合成：`[基础] 文本转音频` - 适合将生成的文本转换为自然语音
  - 视频分析：`[高级] 视频理解` - 适合视频内容分析、视频转文本等任务
  - 多模态任务：`[高级] 全模态整合` - 适合同时处理多种输入的复杂场景

**使用场景示例**：
- 场景1：生成图片描述 → 选择 `[基础] 图像理解` + 连接图片输入
- 场景2：语音识别并生成摘要 → 选择 `[基础] 音频转文本` + 连接音频输入
- 场景3：视频内容分析 → 选择 `[高级] 视频理解` + 连接视频输入
- 场景4：多模态对话 → 选择 `[高级] 全模态整合` + 连接多种输入

#### 提示词配置

参数名：**预设模板 (preset_prompt)**
- 作用说明：选择预设提示词模板，提供针对不同任务的优化提示词结构
- 推荐设置：根据具体任务选择合适的模板
- 常用模板选择指南：
  - 图片描述："[普通] 描述" - 生成简洁明了的图片描述

## 3. Llama-cpp 参数设置节点 (llama_cpp_parameters)

### 功能
设置 LLM 推理的详细参数，控制生成文本的质量和风格。

### 参数说明

参数名：**最大token数 (max_tokens)**
- 调节范围：0-4096
- 作用说明：最大生成token数，影响输出文本长度
- 推荐设置：低性能设备：512-1024，高性能设备：1024-2048
- 实用建议：
  - 简短回答：256-512
  - 详细描述：768-1024
  - 长文本生成：1024-2048

参数名：**采样候选数 (top_k)**
- 调节范围：0-1000
- 作用说明：采样候选数，值越小生成越集中
- 推荐设置：低性能设备：20，高性能设备：30
- 影响分析：
  - 低 top_k（<20）：生成更确定，适合事实性任务
  - 高 top_k（>50）：生成更多样，适合创造性任务

参数名：**核心采样阈值 (top_p)**
- 调节范围：0.0-1.0
- 作用说明：核心采样阈值，控制生成多样性
- 推荐设置：0.85-0.9
- 效果对比：
  - 低 top_p（<0.8）：生成更保守，准确性高
  - 高 top_p（>0.9）：生成更开放，多样性强

参数名：**最小采样概率 (min_p)**
- 调节范围：0.0-1.0
- 作用说明：最小采样概率，避免完全忽略低概率词汇
- 推荐设置：0.05
- 适用场景：需要增加生成多样性时使用

参数名：**典型采样阈值 (typical_p)**
- 调节范围：0.0-1.0
- 作用说明：典型采样阈值，控制生成的典型性
- 推荐设置：1.0

参数名：**生成温度 (temperature)**
- 调节范围：0.0-2.0
- 作用说明：生成温度，值越高越随机
- 推荐设置：0.6-0.8
- 温度指南：
  - 低温度（0.1-0.4）：适合需要准确答案的场景，如问答、代码生成
  - 中温度（0.5-0.8）：适合大多数场景，平衡准确性和创造性
  - 高温度（0.9-1.5）：适合创造性任务，如故事生成、诗歌创作

参数名：**重复惩罚 (repeat_penalty)**
- 调节范围：0.0-10.0
- 作用说明：重复惩罚，避免生成重复内容
- 推荐设置：1.0
- 调整建议：
  - 当出现重复内容时，逐步增加到1.1-1.3
  - 不宜过高，否则可能导致生成内容不连贯

参数名：**频率惩罚 (frequency_penalty)**
- 调节范围：0.0-1.0
- 作用说明：频率惩罚，减少高频词汇出现
- 推荐设置：0.0
- 使用场景：需要减少高频词重复时使用

参数名：**存在惩罚 (presence_penalty)**
- 调节范围：0.0-2.0
- 作用说明：存在惩罚，鼓励生成新内容
- 推荐设置：1.0
- 使用场景：
  - 长文本生成时避免主题漂移
  - 对话中鼓励提出新观点

参数名：**Mirostat模式 (mirostat_mode)**
- 调节范围：0-2
- 作用说明：Mirostat采样模式：0=关闭，1=基础版，2=版本2
- 推荐设置：0
- 高级用户建议：尝试模式1或2以获得更一致的生成质量

参数名：**Mirostat学习率 (mirostat_eta)**
- 调节范围：0.0-1.0
- 作用说明：Mirostat学习率
- 推荐设置：0.1

参数名：**Mirostat目标困惑度 (mirostat_tau)**
- 调节范围：0.0-10.0
- 作用说明：Mirostat目标困惑度
- 推荐设置：5.0

参数名：**对话状态ID (state_uid)**
- 调节范围：-1-999999
- 作用说明：对话状态ID，-1=使用节点唯一ID
- 推荐设置：-1
- 会话管理：
  - 使用不同的state_uid可以同时维护多个独立会话
  - 连续对话时保持相同的state_uid以维持上下文

### 参数调节技巧

1. **控制输出长度**：调整 `max_tokens`，值越大输出越长，但占用更多资源。建议根据实际需求设置，避免设置过大导致性能问题

2. **控制随机性**：`temperature` 是最常用的参数之一，越低越确定（适合需要准确答案的场景），越高越有创意（适合生成故事、诗歌等创造性内容）

3. **控制多样性与准确性平衡**：
   - `top_p` 和 `top_k` 通常配合使用
   - 降低 `top_k` 并提高 `top_p` 可以在保证一定多样性的同时提高准确性
   - 提高 `top_k` 并降低 `top_p` 可以增加生成的多样性

4. **避免重复内容**：
   - 增加 `repeat_penalty` 可以减少重复内容
   - 结合 `presence_penalty` 使用可以鼓励模型生成新内容

5. **参数优先级建议**：
   - 优先调整：`temperature`、`max_tokens`、`top_p`/`top_k`
   - 次要调整：`repeat_penalty`、`presence_penalty`
   - 高级用户：`min_p`、`mirostat` 相关参数

6. **常用场景参数设置**：
   - **事实问答**：temperature=0.1-0.3, top_k=10, top_p=0.7
   - **创意写作**：temperature=0.8-1.2, top_k=50, top_p=0.95
   - **代码生成**：temperature=0.2-0.4, top_k=20, top_p=0.8
   - **对话交互**：temperature=0.6-0.8, top_k=30, top_p=0.9

## 6. 常见问题与解决方案

> 工作流推荐单独分块运行生成，避免提示词生成+图片生成等组合工作流运行，导致资源占用高

### 1. 模型加载失败

**原因**：
- 模型文件不存在或路径错误
- llama-cpp-python 版本过低
- 缺少对应的 mmproj 文件（仅视觉模型）
- 模型格式不兼容

**解决方案**：
- 检查模型文件路径是否正确
- 更新 llama-cpp-python 到最新版本
- 确保 mmproj 文件与模型匹配（仅视觉模型）
- 确认模型格式为 .gguf 或 .safetensors

### 2. 显存不足 (OOM)

**原因**：
- 模型太大，超过显存容量
- 上下文长度设置过大
- 同时运行多个大型模型
- 图片分辨率过高

**解决方案**：
- 降低 `n_gpu_layers` 值
- 减少 `n_ctx` 值
- 使用更小的模型
- 降低图片 `max_size` 值
- 切换到 CPU 模式

### 3. 推理速度慢

**原因**：
- 模型太大
- GPU 层数设置过低
- 硬件性能限制
- system_prompts 字数过多
- 图片分辨率过高

**解决方案**：
- 使用更小的模型
- 增加 `n_gpu_layers` 值
- 降低 `n_ctx` 值
- 关闭不必要的应用程序
- 限制提示词字数，如 300 字以内
- 降低图片分辨率

### 4. 生成结果质量差

**原因**：
- 模型选择不当
- 提示词质量差
- 参数设置不合理
- 模型训练数据限制

**解决方案**：
- 使用更适合任务的模型
- 优化提示词，提供更详细的指令
- 调整 temperature、top_p 等参数
- 使用适合的提示词模板
- 对于复杂任务，考虑使用更强大的模型

### 5. 音频处理失败

**原因**：
- 音频格式不支持
- ASR/TTS 模型选择不当
- 音频文件损坏
- 参考音频格式不正确（TTS音色克隆）

**解决方案**：
- 选择推荐的 ASR 模型
- 检查音频文件是否完整
- TTS音色克隆参考音要求：WAV格式，3~60秒最佳，单声道48000Hz

### 6. 模型加载错误

**错误信息**："模型加载失败"或"不支持的模型类型"

**原因**：
- 模型文件不存在或路径错误
- llama-cpp-python版本过低
- 缺少必要的依赖库
- 模型格式不兼容

**解决方案**：
- 检查模型文件路径是否正确
- 更新llama-cpp-python到最新版本
- 确保安装了所有必要的依赖
- 确认模型格式为.gguf或.safetensors

## 7. 使用建议与最佳实践

### 工作流优化建议

1. **分块运行**：将复杂工作流分成多个简单工作流，避免资源占用过高
2. **及时卸载**：使用 force_offload 参数在不需要时释放显存
3. **状态管理**：使用 save_states 和 state_uid 管理多轮对话
4. **批量处理**：对于大量图片，考虑分批处理

### 提示词优化建议

1. **具体明确**：提供详细的指令和上下文
2. **格式清晰**：使用标点符号和换行提高可读性
3. **示例引导**：提供期望输出的示例
4. **语言一致**：提示词语言与期望输出语言保持一致

### 实际使用案例

#### 案例1：图片描述与提示词生成

**适用场景**：为AI绘画生成详细的提示词

**配置步骤**：
1. **模型选择**：使用 Qwen2.5-Omni-3B 或 Moondream2
2. **推理模式**：选择 `[基础] 图像理解`
3. **预设模板**：选择 `[提示词风格] 详细`
4. **系统提示词**："You are a professional prompt engineer for AI image generation. Create detailed, vivid prompts that will produce high-quality images."
5. **文本输入**："基于这张图片，生成适合SDXL模型的详细提示词，要求风格写实，光线自然。"
6. **参数设置**：temperature=0.7, max_tokens=1024

**预期效果**：生成详细的、适合AI绘画的提示词

#### 案例2：视频内容分析

**适用场景**：分析视频内容并生成摘要

**配置步骤**：
1. **模型选择**：使用 Qwen2.5-Omni-7B 或更高版本
2. **推理模式**：选择 `[高级] 视频理解`
3. **预设模板**：选择 `[视频分析] 视频 - 详细分镜分析`
4. **系统提示词**："You are a professional video analyst. Analyze video content and provide comprehensive summaries of key events and visual elements."
5. **文本输入**："分析这个视频的内容，重点关注主要事件和视觉效果。"
6. **视频参数**：video_max_frames=32, video_sampling=自动均匀采样
7. **参数设置**：temperature=0.6, max_tokens=2048

**预期效果**：生成详细的视频内容分析和摘要

#### 案例3：多模态对话

**适用场景**：同时处理文本、图像和音频的复杂对话

**配置步骤**：
1. **模型选择**：使用 Qwen3.5 或 Qwen2.5-Omni-7B
2. **推理模式**：选择 `[高级] 全模态整合`
3. **预设模板**：选择 `空 - 无`
4. **系统提示词**："You are an excellent multimodal assistant. You can understand and respond to text, images, and audio inputs."
5. **文本输入**：根据具体对话内容输入
6. **参数设置**：temperature=0.7, max_tokens=1024, save_states=True

**预期效果**：能够同时理解和处理多种输入模态的对话

### 性能优化最佳实践

1. **显存管理**：
   - 定期检查显存使用情况
   - 大模型推理后及时使用 force_offload 释放显存
   - 避免同时运行多个大型模型

2. **推理速度优化**：
   - 使用 GPU 模式获得最佳性能
   - 调整 n_gpu_layers 充分利用 GPU 资源
   - 选择合适的批量处理参数

3. **质量优化**：
   - 根据任务类型调整 temperature 参数
   - 使用适合的提示词模板和系统提示词
   - 提供详细的上下文信息

4. **故障排除**：
   - 遇到显存不足时，尝试降低 n_ctx 或切换到 CPU 模式
   - 推理速度慢时，检查 GPU 使用率和驱动版本
   - 生成质量差时，优化提示词和参数设置
